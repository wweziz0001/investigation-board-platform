## 📋 الخطوة الأولى: اقرأ الملفات الأساسية

عند بدء الجلسة، اقرأ الملفات التالية بالترتيب:

```
1. VERSION                    ← لمعرفة الإصدار الحالي
2. DEVELOPMENT_GUIDE.md       ← لفهم منهجية العمل
3. changelog/CHANGELOG.md     ← لمعرفة آخر التغييرات
4. آخر ملف في changelog/v*.md ← لمعرفة تفاصيل آخر إصدار
```

## 🔄 ماذا تفعل عند أي تغيير؟

عندما تقوم بإصلاح مشكلة أو إضافة ميزة أو أي تغيير في الكود، **يجب عليك**:

### 1. تحديث رقم الإصدار

```bash
# اقرأ الإصدار الحالي
cat VERSION
# إذا كان 1.2.6 وكنت تصلح مشكلة ← اجعله 1.2.7
# إذا كنت تضيف ميزة ← اجعله 1.3.0
# إذا كان تغيير جذري ← اجعله 2.0.0
```

### 2. إنشاء ملف التغييرات

أنشئ ملفاً جديداً في `changelog/vX.X.X.md`:

```markdown
# [X.X.X] - YYYY-MM-DD

## 🔧 الإصلاحات / ✨ الميزات الجديدة

### عنوان التغيير

**المشكلة:** (إذا كان إصلاح)
وصف المشكلة

**السبب:**
السبب الجذري

**الحل:**
وصف الحل

**الملفات المتأثرة:**
- `path/to/file.ts` - وصف

**النتيجة:**
- ✅ نتيجة 1
- ✅ نتيجة 2
```

### 3. تحديث CHANGELOG.md

أضف قسم جديد في بداية الملف:

```markdown
## [X.X.X] - YYYY-MM-DD

### 🔧 الإصلاحات

#### عنوان الإصلاح
**المشكلة:** ...
**الملفات المتأثرة:** ...
**التفاصيل:** انظر `changelog/vX.X.X.md`

---
```

### 4. إنشاء فرع Git ورفعه

```bash
# إنشاء فرع
git checkout -b vX.X.X

# إضافة الملفات
git add .

# عمل commit
git commit -m "Release vX.X.X: وصف مختصر"

# رفع إلى GitHub
git push origin vX.X.X
```

## ⚠️ قواعد إلزامية

### ✅ يجب عليك:

1. **دائماً** تحديث `VERSION` عند أي تغيير
2. **دائماً** إنشاء ملف في `changelog/` للإصدار الجديد
3. **دائماً** تحديث `CHANGELOG.md` الرئيسي
4. **دائماً** إنشاء فرع جديد باسم الإصدار
5. **دائماً** توثيق التغييرات بالعربية مع الرموز (🔧 ✨ 🎨 ⚡)

### ❌ لا يجب عليك:

1. عمل commit على فرع main مباشرة
2. تخطي التوثيق
3. حذف ملفات changelog
4. تغيير VERSION بدون تغيير فعلي في الكود
